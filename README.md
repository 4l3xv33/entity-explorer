# Entity Explorer

https://4l3xv33.github.io/entity-explorer/

Static GitHub Pages app for searching client-fetchable restricted entity lists in `data.json`.

## How It Works

This app has no backend and no build step.

- `index.html` loads the page.
- `app.js` loads `data.json`.
- The browser fetches public XML directly from eCFR and OFAC's Sanctions List Service API.
- The browser also reads local snapshots generated by GitHub Actions where public sources do not support direct browser fetches.
- The browser parses XML, CSV, and snapshot HTML tables, then searches records in memory.

All search happens in memory in the browser.

## Current Data Sources

### eCFR

The app uses eCFR where the list data is actually published inside Title 15, Part 744.

- Metadata:
  `https://www.ecfr.gov/api/versioner/v1/titles.json`
- XML:
  `https://www.ecfr.gov/api/versioner/v1/full/{date}/title-15.xml?part=744`
- Parsed lists:
  - A: `Supplement No. 4 to Part 744`, BIS Entity List
  - B: `Supplement No. 7 to Part 744`, BIS Military End User List
  - L: `Supplement No. 6 to Part 744`, Unverified List

### OFAC Sanctions List Service API

The app uses OFAC's documented Sanctions List Service API for browser-fetchable entity XML.

- API documentation:
  `https://sanctionslistservice.ofac.treas.gov/api/PublicationPreview/exports/APIDocumentation.docx`
- C: OFAC SDN List:
  `https://sanctionslistservice.ofac.treas.gov/entities?list=SDN%20List`
- J: EO 14032 / Chinese Military-Industrial Complex sanctions:
  `https://sanctionslistservice.ofac.treas.gov/entities?program=CMIC-EO13959`

The OFAC API response includes a warning that this data may include historical list data and should not be used as the official active list for transaction screening. Entity Explorer uses this API for search and discovery; use OFAC's official list publications for authoritative screening.

### Local Snapshots

Some useful government files are machine-readable but not reliably browser-fetchable from GitHub Pages. For those, the repository can store generated snapshots under `data/snapshots/`.

- D: BIS Denied Persons List:
  `data/snapshots/dpl.csv`
- F: DDTC Debarred Parties List:
  `data/snapshots/ddtc-debarred-statutory.csv`
  `data/snapshots/ddtc-debarred-administrative.csv`
- G: FCC Covered List table:
  `data/snapshots/fcc-coveredlist.html`

The DPL snapshot is generated by `.github/workflows/update-dpl-snapshot.yml`, which reads the BIS DPL page, discovers the current `dpl_*.csv` file, and writes:

- `data/snapshots/dpl.csv`
- `data/snapshots/dpl-source-url.txt`

The DDTC snapshot is generated by `.github/workflows/update-ddtc-debarred-snapshot.yml`, which reads the DDTC ServiceNow article JSON, discovers the current CSV attachment IDs, downloads the statutory and administrative CSV files, normalizes them to UTF-8, and writes:

- `data/snapshots/ddtc-debarred-statutory.csv`
- `data/snapshots/ddtc-debarred-administrative.csv`
- `data/snapshots/ddtc-debarred-source-urls.txt`

The FCC snapshot is generated by `.github/workflows/update-fcc-coveredlist-snapshot.yml`, which downloads the FCC Covered List page HTML with Node's browser-compatible `fetch()` and writes:

- `data/snapshots/fcc-coveredlist.html`
- `data/snapshots/fcc-coveredlist-source-url.txt`

The workflows run daily and can also be started manually from the GitHub Actions tab.

## What This Can and Cannot Do

This static app can use sources that meet both conditions:

- The source publishes machine-readable data, preferably XML, JSON, CSV, or a stable HTML table.
- The source can be fetched from a browser running on GitHub Pages.

Currently supported:

- A: BIS Entity List
- B: BIS Military End User List
- C: OFAC SDN List
- D: Denied Persons List from local CSV snapshot
- F: DDTC Debarred Parties List from local CSV snapshots
- G: PRC Telecommunications Companies List from FCC Covered List HTML snapshot
- J: EO 14032 / CMIC sanctions from OFAC Sanctions List Service API
- L: Unverified List

Currently not supported client-side:

- E: Chinese Military Companies List
- H: Military-Civil Fusion Affiliated Institutions List
- I: PRC Semiconductor Companies List
- K: FCC Covered List
- M: UFLPA Entity List
- N: Biotechnology Company of Concern List
- O: Catch-all provision

Those may require a backend fetcher, a generated static data snapshot, PDF parsing, portal-specific handling, or a source-specific API that is not currently wired into this app.

## BIS Denied Persons List Observation

The BIS Denied Persons List page is:

- `https://www.bis.gov/licensing/end-user-guidance/denied-persons-list-dpl`

That page embeds a direct CSV data source. At the time this was wired up, the discovered CSV was:

- `https://media.bis.gov/sites/default/files/dpl_04142026.csv`

The CSV has useful columns including `Name_and_Address`, `Effective_Date`, `Expiration_Date`, `Appropriate Federal Register Citations`, and `Type of Denial`. It is a good candidate for integration, but the CSV response does not currently advertise browser CORS headers, so it is not safe to assume a GitHub Pages browser can fetch it directly. The likely static-site approach is to snapshot this CSV into the repository with a scheduled GitHub Action.

This app now uses that snapshot approach.

## DDTC Debarred Parties List Observation

The DDTC Debarred Parties page is:

- `https://www.pmddtc.state.gov/ddtc_public?id=ddtc_kb_article_page&kb_number=KB0010126`

The public page is a ServiceNow portal article. Its static HTML does not directly expose the useful data, but the portal JSON endpoint does:

- `https://www.pmddtc.state.gov/api/now/sp/page?id=ddtc_kb_article_page&kb_number=KB0010126`

That JSON includes downloadable CSV attachment links for both statutory and administrative debarments. At the time this was wired up, the discovered CSV attachments were:

- Statutory: `https://www.pmddtc.state.gov/sys_attachment.do?sys_id=15370fe2977ba21067b1791ad053af0d`
- Administrative: `https://www.pmddtc.state.gov/sys_attachment.do?sys_id=d78bbc2f1b8f29d0c6c3866ae54bcbd7`

The static app reads checked-in UTF-8 snapshots of those CSVs. The GitHub Action re-discovers the attachment IDs from the article JSON on each run so it can follow updates if DDTC replaces the attachment files.

## FCC Covered List Observation

The FCC Covered List page is:

- `https://www.fcc.gov/supplychain/coveredlist`

The useful data is published as a static HTML table headed `Covered List`. This repository snapshots the full page HTML and the browser parses the first Covered List table with `DOMParser`.

The local `curl` command can be unreliable against this FCC page, but Node's built-in `fetch()` retrieved the HTML successfully. The GitHub Action uses that Node fetch path rather than curl.

The table is broader than only PRC telecommunications companies because it is the FCC Covered List page. Entity Explorer currently wires it to SIRA § 2(e)(3)(G) because that item points to this FCC page in `data.json`; SIRA § 2(e)(3)(K) also points to the same FCC Covered List and can reuse the same snapshot later.

## OFAC Observations

OFAC does publish machine-readable XML downloads for the SDN List and consolidated sanctions data:

- `https://www.treasury.gov/ofac/downloads/sdn.xml`
- `https://www.treasury.gov/ofac/downloads/consolidated/consolidated.xml`

An attempted browser-only integration was retracted because the OFAC download path redirects through `sanctionslistservice.ofac.treas.gov` to signed S3 URLs. That can work from command-line tools, but it is not reliable from GitHub Pages because browser `fetch()` enforces CORS across the redirect chain.

The documented Sanctions List Service `/entities` API is different from those download URLs and is currently used by the app. It sends browser-accessible CORS headers, but its response includes OFAC's historical-data warning and the SDN endpoint is large, so it can take time to load.

If the browser-only API path becomes unreliable, the fallback approach is to use a scheduled GitHub Action to download OFAC XML into this repository, then have the static app read those checked-in snapshots locally.

## Notes

The app marks supported sources as `Live eCFR`, `Live OFAC API`, or `Live Snapshot`. The other sources remain visible but are marked `Not client-fetchable` because they are not currently wired to a browser-fetchable machine-readable source or local snapshot.

Opening `index.html` directly from disk may fail in some browsers because `fetch("data.json")` is restricted under `file://`. GitHub Pages works because it serves the files over HTTPS.
